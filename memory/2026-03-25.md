- OpenClaw launcher/login failure was caused by `start-openclaw.bat` preferring a missing `.local\openclaw\openclaw.json` path and then not cleanly falling back to `E:\Auto\projects\openclaw.json`.
- The fixed startup flow now:
  - calls `start-openclaw.local.bat` first,
  - uses the local config only if it actually exists,
  - otherwise falls back to `E:\Auto\projects\openclaw.json`,
  - and then launches `openclaw gateway run --port 5000 --force --token ...`.
- The launcher is considered healthy again when `Test-NetConnection 127.0.0.1 -Port 5000` returns `True` and Edge opens `http://127.0.0.1:5000/control-ui-custom/launch.html#token=...&gatewayUrl=ws://127.0.0.1:5000`.
- `start-openclaw.local.bat` now carries the local token plus the COZE env placeholders needed to keep the gateway from exiting early on config validation.
- Experience Manager has been cut back to cloud-primary storage again. The runtime now reads the existing MySQL tables `experience_records_cloud` and `experience_notes_cloud`, the local `experiences.json` and `notes.json` files were deleted, and the repo no longer depends on `EXPERIENCE_DATA_PATH` / `NOTE_DATA_PATH` as startup defaults.
- The cloud database is reachable at `124.220.245.121:22295`, and after the cutover the cloud tables contain 125 experience records and 43 notes.
- The `OpenClaw UI` desktop shortcut now points to `E:\Auto\start-openclaw-admin.ps1`, which self-elevates with UAC before launching the normal startup chain. This is the practical "always admin" path for OpenClaw; Windows still requires the user to accept the elevation prompt.
- OpenClaw qwen restore: the real breakage was a malformed `~\.openclaw\openclaw.json` / `projects\openclaw.json` pair (BOM plus a trailing comma in `models.providers`) combined with stale session state. Fix was to add an explicit `ollama` provider for `qwen2.5:7b`, remove the stale `agent:main:main` session mapping by resetting `C:\Users\ASUS-KL\.openclaw\agents\main\sessions\sessions.json` to `{}`, and restart the gateway.
- Ollama `qwen2.5:7b` later failed with `unable to allocate CUDA0 buffer`; the practical workaround was to force `num_gpu: 0` in `ai-model-hub`'s Ollama request body so the model stays on CPU instead of trying to offload into VRAM.
- The CPU-only fallback still needed stricter load settings to succeed on this machine: `num_gpu: 0`, `num_ctx: 1024`, `num_batch: 1`, `low_vram: true`, and `use_mmap: true` together made `/api/generate` return `Pong!` instead of `unable to allocate CPU buffer`.
- The real reason OpenClaw still failed after the plugin-side low-VRAM fix was that its own provider config kept injecting `num_ctx` from a `contextWindow` of 32768. Dropping `models.providers.ollama.models[0].contextWindow` and the mirrored `anthropic` alias to 1024 (with `maxTokens: 2048`) was necessary so OpenClaw stops over-allocating Ollama before the request even reaches the plugin layer.
- OpenClaw's own agent layer has a hard minimum context window requirement of 16000 tokens. Dropping the model config to 1024 fixed memory pressure but immediately broke agent startup with `Model context window too small (1024 tokens). Minimum is 16000.` The practical floor for this stack is therefore 16000, not lower; keep low-VRAM settings in the Ollama request body instead of shrinking `contextWindow` below the agent minimum.
- The remaining `CUDA_Host buffer` failures were not just about context length. On this machine the runner also needed a much smaller `maxTokens` budget, so the final stable profile is `contextWindow: 16000` with `maxTokens: 512` (plus the low-VRAM/Ollama CPU settings). That keeps OpenClaw's agent minimum satisfied while reducing the runtime allocation pressure enough for qwen2.5:7b to load.
