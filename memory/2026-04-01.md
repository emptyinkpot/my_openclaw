# 2026-04-01

- Key Guard 这次故障被抽象为“三层叠加”问题：执行层表现为页面跳错，模板层缺少统一验收入口，结构层存在 canonical/alias/fallback 职责混淆。
- 真实根因链路固定为：网关运行态未加载新路由 + `vscode-key-guard` 插件注册失败（含 ESM 导入与缺文件风险）+ `control-ui-custom` 旧路由与 fallback 壳页掩盖真实失败。
- 固定验收顺序沉淀为硬门槛：`/health` -> `/api/<plugin>/health` -> canonical 页面内容校验 -> alias 302 校验 -> 导航单一源一致性校验。
- 路由治理原则：canonical path 只能有一个；兼容路径只做跳转不承载真实页面；壳层不得吞错，至少在开发态显示 fallback 命中与目标插件 health 状态。
- 插件治理原则：插件“可加载”与“可写入”分层；写链路依赖缺失时必须 fail-closed 且状态显式，不得伪装成功；每个插件必须提供 `/api/<module>/health`。
- 导航治理原则：共享导航必须单一来源（SSOT），禁止多份人工镜像并行维护；若保留镜像副本，必须脚本单向同步并可校验漂移。
- 启动链治理原则：区分“网关已启动”与“插件已就绪”；启动脚本通过不等于模块就绪，发布前必须做插件 load-check + route-verify。
- 本事件建议沉淀为标准入口（脚本/任务）：`ai:gateway:health`、`ai:plugin:health`、`ai:route:verify`、`ai:plugin:load-check`、`ai:nav:sync-check`，用于替代人脑记忆式排障。
