- The last UI black-screen regression came from accidental corruption inside the minified `projects/control-ui-custom/assets/index-UvgeZ3yV.js` bundle itself, not from missing files. The exact breakage was a malformed icon map entry plus a few damaged template interpolations (`${...}` became `�{...}`), which caused browser parse errors and kept `openclaw-app` from registering. The reliable fix was to use a parser (`acorn`) to pinpoint the syntax break, patch the smallest possible characters, mirror the fix into the installed `openclaw/dist/control-ui` copy, and then re-check the page via remote debugging until `customElements.get("openclaw-app")` returned `true`.
- The remaining UI text corruption was not in the main nav HTML after all; the actual served `/shared/nav-bar.html` path under `control-ui-custom/shared` had to be updated, then `bootstrap.js` and `nav-bar-behavior.js` needed cache-busting version bumps. After that, the only leftover visible glitch was a single breadcrumb separator inside `dashboard-header__breadcrumb-sep`, which was corrected by changing the broken text node back to `>`. The lesson: inspect the real DOM and the real served static path first, then fix cache/versioning, and only then chase any leftover bundle text nodes one by one.

- The shortcut launch regression was caused by two separate startup hazards: `start-openclaw.local.bat` had a UTF-8 BOM that made `cmd` misread its first line, and the `OpenClaw Gateway` scheduled task could stay registered/running long enough to make a fresh `openclaw gateway run` hit a lock timeout. The practical fix was to rewrite the local `.bat` without BOM, end the scheduled task at startup, and open the browser immediately after launching the gateway instead of waiting on the slow `gateway status` health probe.
