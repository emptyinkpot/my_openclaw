# 技术决策记录 (ADR)

## ADR-001: 使用 Playwright 进行浏览器自动化

**状态**: 已接受  
**日期**: 2026-03-12  
**决策者**: OpenClaw 配置专家

### 背景

需要自动化操作白梦写作网，实现：
- 自动登录
- 获取作品列表
- 进入作品详情

### 考虑方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Playwright** | 功能强大、支持多浏览器、自动等待 | 资源占用较高 |
| Puppeteer | Chrome 原生支持、社区活跃 | 仅支持 Chrome |
| Selenium | 历史悠久、语言支持广泛 | 配置复杂、稳定性差 |
| API 直接调用 | 性能好、资源占用低 | 需要逆向工程、可能违反 ToS |

### 决策

选择 **Playwright** 作为浏览器自动化工具。

**理由**:
1. 支持自动等待，减少 flaky 测试
2. 内置截图、录制功能，便于调试
3. 支持 storageState，方便登录状态持久化
4. 社区活跃，文档完善

### 后果

**正面**:
- 开发效率高
- 调试方便（截图、录像）
- 跨浏览器支持（未来可扩展）

**负面**:
- 需要安装浏览器，占用磁盘空间
- 内存占用较高

## ADR-002: 使用 storageState 持久化登录状态

**状态**: 已接受  
**日期**: 2026-03-12  
**决策者**: OpenClaw 配置专家

### 背景

白梦写作网使用微信扫码登录，需要避免每次操作都重新扫码。

### 考虑方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **storageState** | Playwright 原生支持、包含 Cookie 和 LocalStorage | 需要定期刷新 |
| 手动提取 Token | 灵活控制 | 需要处理多种存储机制 |
| 用户名密码登录 | 简单直接 | 网站不支持 |

### 决策

使用 Playwright 的 `context.storageState()` 保存登录状态。

**实现细节**:
1. 登录成功后调用 `context.storageState({ path: stateFile })`
2. 初始化时通过 `storageState: stateFile` 加载状态
3. 状态文件保存 JWT Token（在 LocalStorage 中）

### 后果

**正面**:
- 实现简单，无需手动处理 Cookie
- 支持多种存储机制（Cookie、LocalStorage、SessionStorage）

**负面**:
- Token 有过期时间，需要定期重新登录
- 状态文件需要妥善保管（包含敏感信息）

## ADR-003: 重构为通用网站自动化模块

**状态**: 已接受  
**日期**: 2026-03-12  
**决策者**: OpenClaw 配置专家

### 背景

原有的 `baimeng-writer` 技能是专门为白梦写作网设计的，代码耦合度高，难以扩展到其他网站。

### 考虑方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **通用模块** | 可复用、易于扩展、配置驱动 | 初期设计复杂 |
| 复制粘贴修改 | 快速实现 | 维护困难、代码重复 |
| 抽象基类 | 结构清晰 | 学习成本高 |

### 决策

创建 `WebsiteAutomation` 通用模块，通过配置文件支持多网站。

**设计原则**:
1. 配置驱动：网站特定的选择器、URL 等通过配置文件定义
2. 核心类通用：自动化逻辑（登录、截图、状态管理）抽象为通用类
3. CLI 接口统一：所有网站使用相同的命令行接口

### 后果

**正面**:
- 新增网站只需添加配置，无需修改代码
- 核心逻辑集中维护，修复 Bug 更容易
- 测试覆盖更全面

**负面**:
- 初期设计需要考虑更多边界情况
- 配置复杂度增加

## ADR-004: 单向文档同步（代码→文档）

**状态**: 已接受  
**日期**: 2026-03-12  
**决策者**: OpenClaw 配置专家

### 背景

需要建立文档系统记录项目信息、Bug 修复经验等，考虑是否采用双向同步。

### 考虑方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **单向同步** | 安全可靠、实现简单 | 文档不能直接修改代码 |
| 双向同步 | 灵活性高 | 可能导致代码被意外覆盖 |
| 手动维护 | 完全可控 | 工作量大、容易遗漏 |

### 决策

采用 **单向同步**（代码→文档）。

**实现方式**:
1. 代码变更检测脚本监控文件变化
2. 自动生成 API 文档、脚本清单
3. Bug 记录、技术决策等手动维护
4. 提供手动触发文档生成的命令

### 后果

**正面**:
- 代码安全，不会被文档覆盖
- 自动化程度高，减少维护工作
- 文档与代码保持一致

**负面**:
- 需要通过修改代码来更新文档
- 非代码类文档（如经验总结）仍需手动维护
