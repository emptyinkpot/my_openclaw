# 插件架构设计 - 合并 vs 分开的决策指南

## Summary
插件架构设计 - 合并 vs 分开的决策指南

## Content
## 两个插件的功能定位

### novel-manager（小说管理插件）
- 作品管理（增删改查）
- 番茄小说扫描和发布
- 缓存管理
- 自动化流水线
- 飞书集成
- 技术特点：重型（MySQL + Playwright）

### experience-manager（经验积累插件）
- 记录解决问题的经验
- 经验统计和分析
- 内存同步
- 技术特点：轻量级（JSON 文件）

## 合并的优缺点

### 合并的优点
1. 统一入口 - 一个插件管所有
2. 共享资源 - 共享数据库连接、工具函数
3. 统一配置 - 只维护一个插件配置

### 合并的缺点
1. 职责混乱 - 两个完全不同的业务领域
2. 耦合度高 - 修改一个影响另一个
3. 难以独立升级 - 必须一起更新
4. 插件变得庞大 - 难以维护
5. 违反单一职责原则

## 分开的优缺点

### 分开的优点
1. 职责清晰 - 每个插件只做自己的事
2. 低耦合 - 互不影响
3. 可独立发布 - 单独更新、启用/禁用
4. 可复用 - 经验积累插件可在其他项目复用
5. 符合 OpenClaw 设计理念
6. 便于测试

### 分开的缺点
1. 共享资源麻烦
2. 配置分散
3. 可能路由冲突

## 关键判断标准

| 问题 | 是 | 否 |
|------|----|----|
| 两个功能会被不同的人使用吗？ | 分开 | 合并 |
| 想单独启用/禁用某个功能吗？ | 分开 | 合并 |
| 业务领域逻辑上独立吗？ | 分开 | 合并 |
| 未来可能移到其他项目吗？ | 分开 | 合并 |
| 共享很多代码吗？ | 合并 | 分开 |

## 最终建议

**保持分开！**

理由：
1. 业务领域完全不同
2. 技术栈差异大
3. 符合插件化架构理念
4. 未来可扩展性好

## 总结对比

| 维度 | 分开 | 合并 |
|------|------|------|
| 维护成本 | 低 | 高 |
| 可扩展性 | 好 | 一般 |
| 职责清晰 | 清晰 | 混乱 |
| 复用性 | 高 | 低 |
