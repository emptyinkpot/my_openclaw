## 第一章：浏览器是如何工作的

### 1.1 浏览器的两种"面孔"

想象浏览器是一个人：

**有头模式 (Headed Mode)** = 正常人
- 有眼睛（图形界面）
- 你能看到他在做什么
- 他可以看、可以点击、可以输入

**无头模式 (Headless Mode)** = 蒙眼工作的人
- 没有图形界面
- 你看不到他在做什么
- 但他依然可以完成工作（点击、输入、抓取数据）

```
有头模式：
┌──────────────────────────────┐
│  🌐 浏览器窗口               │  ← 你能看到
│  ┌──────────────────────┐    │
│  │ 网页内容              │    │
│  │  [按钮]  [输入框]     │    │
│  └──────────────────────┘    │
└──────────────────────────────┘

无头模式：
┌──────────────────────────────┐
│  👤 浏览器（看不见）         │  ← 你看不到
│     正在后台运行...          │
│     ✓ 点击完成               │
│     ✓ 数据已抓取             │
└──────────────────────────────┘
```

### 1.2 为什么要用无头模式？

**有头模式的缺点**：
- 需要显示器（在服务器上跑不了）
- 占用更多内存和CPU
- 界面可能被意外操作

**无头模式的优点**：
- ✅ 可以在没有显示器的服务器上运行
- ✅ 速度更快（不需要渲染图形）
- ✅ 节省资源
- ✅ 不会被人误操作

**本项目中的应用**：
```javascript
// config.js 中的配置
const baimengConfig = {
  headless: false,  // 登录时设为 false，方便扫码
  // ...
};

// 登录成功后，后续操作可以设为 true（无头）
```

## 第二章：Playwright 的运行模式详解

### 2.1 基本运行模式

| 模式 | 配置 | 适用场景 |
|------|------|----------|
| **有头模式** | `headless: false` | 开发调试、需要看界面 |
| **无头模式** | `headless: true` | 生产环境、定时任务 |
| **开发者模式** | `devtools: true` | 深度调试、查看网络请求 |
| **慢动作模式** | `slowMo: 1000` | 演示、教学 |

### 2.2 开发者模式 (DevTools)

就像汽车的仪表盘，可以看到引擎内部运转：

```javascript
const browser = await chromium.launch({
  headless: false,
  devtools: true  // 打开开发者工具
});
```

**能看到什么**：
- 网络请求（网页加载了哪些资源）
- Console 日志（JavaScript 报错信息）
- Elements（网页的 HTML 结构）
- Application（Cookie、LocalStorage）

**什么时候用**：
- 调试登录问题
- 查看 API 请求
- 分析网页结构

### 2.3 慢动作模式 (Slow Motion)

让浏览器像慢动作回放一样运行，方便观察：

```javascript
const browser = await chromium.launch({
  headless: false,
  slowMo: 1000  // 每个操作延迟 1000 毫秒（1秒）
});
```

**适用场景**：
- 制作演示视频
- 调试快速操作的问题
- 学习观察自动化过程

### 2.4 移动端模拟 (Mobile Emulation)

让浏览器假装自己是手机：

```javascript
const context = await browser.newContext({
  viewport: { width: 375, height: 667 },  // iPhone 尺寸
  userAgent: 'Mozilla/5.0 (iPhone; CPU iPhone OS...'
});
```

**有什么用**：
- 测试网页的手机版
- 抓取手机专属内容
- 模拟不同设备

## 第三章：计算机基础知识

### 3.1 进程与线程（简单理解）

**进程 (Process)** = 一个独立的程序
- 就像一家独立的公司
- 有自己的办公室（内存空间）
- 与其他公司隔离

**线程 (Thread)** = 公司里的员工
- 同一个公司的员工共享办公室
- 可以同时做多件事
- 需要协调配合

```
浏览器进程：
┌─────────────────────────────────────┐
│  Chrome 浏览器（公司）              │
│  ┌──────────┐  ┌──────────┐        │
│  │ 标签页1  │  │ 标签页2  │        │ ← 线程（员工）
│  │ (员工A)  │  │ (员工B)  │        │
│  └──────────┘  └──────────┘        │
│  ┌──────────┐                      │
│  │ 标签页3  │                      │
│  │ (员工C)  │                      │
│  └──────────┘                      │
└─────────────────────────────────────┘
```

**本项目中的应用**：
- 每个 Playwright 实例是一个进程
- 每个页面是一个线程
- 关闭浏览器 = 关闭整个公司（所有员工都走了）

### 3.2 内存与存储

**内存 (RAM)** = 办公桌
- 临时存放正在用的东西
- 速度快，但断电就清空
- 容量有限

**存储 (硬盘/SSD)** = 文件柜
- 长期保存文件
- 速度慢，但断电不丢
- 容量大

```
程序运行：
┌──────────────────────────────────┐
│  内存（办公桌）                   │
│  ┌─────────┐  ┌─────────┐        │
│  │ 网页数据 │  │ 登录状态 │        │ ← 临时存放
│  └─────────┘  └─────────┘        │
└──────────────────────────────────┘
           ↓ 保存
┌──────────────────────────────────┐
│  存储（文件柜）                   │
│  ┌─────────────────────────────┐ │
│  │ baimeng_state.json          │ │ ← 长期保存
│  │ (登录凭证)                   │ │
│  └─────────────────────────────┘ │
└──────────────────────────────────┘
```

**本项目中的应用**：
- 登录状态先存在内存（运行时的页面）
- 保存后存到硬盘（state.json 文件）
- 下次启动从硬盘读取到内存

### 3.3 网络基础

#### IP 地址 = 门牌号
- 每台联网设备的唯一标识
- 例如：`192.168.1.1`

#### 端口 (Port) = 房间号
- 一台设备可以运行多个服务
- 不同服务用不同端口区分
- 例如：HTTP 用 80，HTTPS 用 443

```
服务器（大楼）：
┌──────────────────────────────────┐
│  IP: 192.168.1.100               │
│  ┌──────────┐  ┌──────────┐      │
│  │ 端口 5000│  │ 端口 8080│      │ ← OpenClaw Gateway
│  │ (Gateway)│  │ (其他服务)│      │
│  └──────────┘  └──────────┘      │
└──────────────────────────────────┘
```

**本项目中的应用**：
- OpenClaw Gateway 运行在 `0.0.0.0:5000`
- `0.0.0.0` 表示接受所有IP的连接
- `5000` 是端口号

### 3.4 Cookie 与 LocalStorage

都是浏览器存储数据的方式：

**Cookie** = 贴在门上的便签
- 每次访问网站都会带过去
- 服务器可以读取
- 有大小限制（4KB）
- 有过期时间

**LocalStorage** = 家里的储物柜
- 只存在浏览器里
- 服务器不能直接读取
- 容量更大（5MB）
- 除非手动删除，否则永久保存

```
浏览器存储：
┌──────────────────────────────────┐
│  baimengxiezuo.com               │
│  ┌────────────────────────────┐  │
│  │ Cookie（便签）              │  │
│  │ • session_id=abc123        │  │
│  │ • expires=明天              │  │
│  └────────────────────────────┘  │
│  ┌────────────────────────────┐  │
│  │ LocalStorage（储物柜）      │  │
│  │ • token=eyJhbG...          │  │ ← JWT Token
│  │ • user_preferences={...}   │  │
│  └────────────────────────────┘  │
└──────────────────────────────────┘
```

**本项目中的应用**：
- 白梦写作网用 **LocalStorage** 存 JWT Token
- 这就是为什么 Playwright 的 `storageState` 要同时保存 Cookie 和 LocalStorage

## 第四章：JWT Token 是什么？

### 4.1 简单理解

**JWT (JSON Web Token)** = 电子身份证

就像你去健身房：
1. 第一次：出示身份证，办理会员卡（登录）
2. 之后：出示会员卡即可进入（Token 验证）

```
登录过程：
┌────────┐      扫码登录       ┌──────────────┐
│ 用户   │ ──────────────────→ │ 白梦写作网    │
└────────┘                     └──────────────┘
                                    │
                                    ↓ 验证成功
                              ┌──────────────┐
                              │ 发放 Token   │
                              │ eyJhbGci...  │
                              └──────────────┘
                                    │
                                    ↓
┌────────┐    携带 Token 访问    ┌──────────────┐
│ 用户   │ ────────────────────→ │ 已登录状态   │
└────────┘                      └──────────────┘
```

### 4.2 Token 的结构

JWT 由三部分组成，用点号分隔：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6IjEyMyJ9.signature
└─────────────────────────────────┘
              │
              ├── 头部 (Header): 算法信息
              ├── 载荷 (Payload): 用户信息
              └── 签名 (Signature): 防篡改
```

**内容示例**：
```json
// 头部
{
  "alg": "HS256",  // 使用的算法
  "typ": "JWT"     // 类型
}

// 载荷（真正有用的信息）
{
  "id": "用户ID",
  "name": "用户名",
  "provider": "wechat",  // 微信登录
  "iat": 1773291686      // 签发时间
}
```

**本项目中的应用**：
```javascript
// 从 LocalStorage 读取的 Token
{
  "name": "token",
  "value": "eyJhbGciOiJIUzI1NiIs..."
}

// 这个 Token 就是登录凭证，下次访问时带着它，网站就知道你是谁
```

## 第五章：命令行与脚本

### 5.1 什么是命令行？

**图形界面 (GUI)** = 用鼠标点点点
**命令行 (CLI)** = 用文字发指令

就像去餐厅：
- GUI：指着菜单上的图片说"我要这个"
- CLI：直接说"来一份宫保鸡丁"

```bash
# 图形界面操作
# [双击] 文件夹 → [双击] 文件

# 命令行操作
# 一句话完成
cat /workspace/projects/docs/README.md
```

### 5.2 常用命令

| 命令 | 作用 | 类比 |
|------|------|------|
| `ls` | 列出文件 | 打开文件夹看里面有什么 |
| `cd` | 切换目录 | 双击进入文件夹 |
| `cat` | 查看文件内容 | 双击打开文件 |
| `mkdir` | 创建文件夹 | 右键新建文件夹 |
| `rm` | 删除文件 | 拖到回收站 |
| `node` | 运行 JavaScript | 双击运行程序 |

### 5.3 什么是脚本？

**脚本 (Script)** = 批量命令清单

就像做菜菜谱：
- 普通命令："切一个土豆"（单次操作）
- 脚本：完整的菜谱（一系列操作）

```javascript
// 这是一个脚本（baimeng_login.js）

// 第1步：打开浏览器
const browser = await chromium.launch();

// 第2步：打开网页
const page = await browser.newPage();
await page.goto('https://www.baimengxiezuo.com');

// 第3步：点击登录
await page.click('text=登录');

// 第4步：保存状态
await context.storageState({ path: 'state.json' });

// 第5步：关闭浏览器
await browser.close();
```

运行脚本：
```bash
node baimeng_login.js
# 上面5步一次性自动完成
```

## 第六章：前后端是什么？

### 6.1 餐厅比喻

**前端** = 餐厅大堂
- 顾客看到的地方
- 菜单、装修、服务员
- 负责展示和交互

**后端** = 厨房
- 顾客看不到的地方
- 厨师、食材、烹饪
- 负责处理业务逻辑

**数据库** = 仓库
- 存放食材（数据）
- 按需要取用

```
完整的网站就像餐厅：

顾客（用户）
    │
    ▼
┌────────────────────────────────────┐
│  前端（大堂）                       │
│  • 网页界面                        │
│  • 按钮、输入框                     │
│  • 展示数据                        │
└──────────────┬─────────────────────┘
               │  HTTP 请求
               ▼
┌────────────────────────────────────┐
│  后端（厨房）                       │
│  • 处理请求                        │
│  • 业务逻辑                        │
│  • 验证权限                        │
└──────────────┬─────────────────────┘
               │ SQL 查询
               ▼
┌────────────────────────────────────┐
│  数据库（仓库）                     │
│  • 存储数据                        │
│  • 用户表、作品表                   │
└────────────────────────────────────┘
```

### 6.2 本项目中的前后端

**前端**：
- 白梦写作网的网页
- Playwright 操作的界面

**后端**：
- OpenClaw Gateway（处理消息）
- 白梦写作网的服务器（处理登录、数据）

**我们的代码位置**：
```
用户（飞书）
    │
    ▼
OpenClaw Gateway（后端）
    │
    ▼
我们的 Skill 代码（自动化脚本）
    │
    ▼
Playwright 控制的浏览器（前端操作）
    │
    ▼
白梦写作网（目标网站）
```

## 第七章：重要概念总结

### 7.1 本项目涉及的技术栈

### 7.2 常见问题速查

**Q: 为什么登录状态会过期？**
A: Token 像临时通行证，通常设置过期时间（比如7天），过期后需要重新登录。

**Q: 无头模式能截图吗？**
A: 可以！无头模式只是不显示窗口，但可以保存截图到文件。

**Q: 为什么需要保存登录状态？**
A: 否则每次运行都要重新扫码，像每次进门都要重新办会员卡。

**Q: LocalStorage 和 Cookie 有什么区别？**
A: Cookie 会自动发给服务器（像名片），LocalStorage 只存在浏览器里（像日记本）。

**Q: 什么是异步操作？**
A: 像点外卖，下单后不必等着，可以先做别的事，做好了会通知你。
