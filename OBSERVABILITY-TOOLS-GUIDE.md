# Atramenti-Console 可观测性与可视化工具指南

## 1. 文档目的

这份文档用于回答一个核心问题：

有没有主流工具，可以可视化看到部署到服务器上的：

- 各个端口
- 各个服务
- 进程与资源占用
- 服务之间的依赖关系
- 请求链路和调用关系

答案是：有，而且主流方案不止一种，但它们看的层次不一样。

## 2. 先明确：你想看的其实是三种不同的东西

### 第一类：主机与端口视角

你想知道：

- 哪个进程占用了哪个端口
- 哪个服务正在运行
- 哪些连接正在建立
- CPU、内存、网络流量如何

这类更偏“服务器机器本身”。

### 第二类：应用与服务依赖视角

你想知道：

- 一个网页请求经过了哪些服务
- 哪个服务调用了哪个 API
- 哪个下游服务慢了
- 服务依赖关系图是什么样

这类更偏“应用调用链”。

### 第三类：部署结构视角

你想知道：

- 哪些服务属于同一套部署
- 它们映射了哪些端口
- 它们挂了哪些卷、网络、环境变量
- 当前部署拓扑是什么样

这类更偏“部署和运维结构”。

## 3. 主流工具怎么分

### 3.1 Netdata

Netdata 更适合：

- 单机服务器
- 云主机
- 快速装上就看机器状态
- 可视化查看进程、端口、网络、磁盘、系统负载

它适合回答的问题：

- 哪个进程占了端口
- 哪个服务 CPU 飙高
- 网络连接多不多
- 当前机器压力在哪里

#### 优点

- 上手快
- 面板直观
- 很适合先看“服务器现在到底发生了什么”
- 不需要你先把应用做很多改造

#### 缺点

- 它不是严格意义上的“应用依赖图”工具
- 对复杂服务调用链的展示不如 OpenTelemetry 体系

#### 对你当前阶段的建议

如果你现在最想看的是：

- 服务器跑了哪些东西
- 端口占用情况
- 服务状态和资源情况

那最适合的起步工具就是 Netdata。

官网：`https://www.netdata.cloud/`

## 4. Grafana + OpenTelemetry

这套更适合看“服务依赖图”。

### 4.1 它能看什么

如果你的应用接入了 OpenTelemetry traces，那么 Grafana 可以展示：

- Service Map（服务地图）
- 请求链路
- 每次请求经过了哪些服务
- 哪个服务延迟高
- 哪个服务是瓶颈

### 4.2 它适合什么场景

适合：

- 网站前端 + 后端 API + 数据存储 + 其他服务
- 有多个模块相互调用
- 你希望从“业务请求”视角理解系统

### 4.3 优点

- 主流
- 专业
- 依赖关系图能力强
- 后续扩展性好

### 4.4 缺点

- 需要把应用接入遥测数据
- 不是装上就自动全懂你的系统
- 初始配置比 Netdata 重

### 4.5 对你当前阶段的建议

当你的项目从“单机跑起来”进入“多个服务长期联动”阶段时，再上这套最值。

也就是说：

- 现在可以先不急着上
- 等网站、OpenClaw、网关、管理端逐渐稳定后再接入

Grafana 相关文档：

- Application Observability
- Service Map

## 5. Portainer

Portainer 更适合容器化部署。

### 5.1 它看什么

它更偏：

- 容器列表
- 容器状态
- 端口映射
- 网络
- 卷
- Stack / Compose 结构

### 5.2 它适合什么场景

适合：

- Docker 部署
- Docker Compose
- 多容器服务管理

### 5.3 优点

- 可视化管理容器非常方便
- 很适合部署之后从“容器拓扑”角度看系统

### 5.4 缺点

- 它不是“应用调用链”工具
- 如果你现在还不是容器化部署，它的价值没那么大

### 5.5 对你当前阶段的建议

如果未来你把网站、OpenClaw、数据库、反向代理都改成 Docker 化部署，那 Portainer 会很实用。

但如果你现在还主要是：

- 一台 Linux 机
- SSH 运维
- 直接跑服务

那它还不是第一优先级。

## 6. 你现在最适合哪一种

结合你当前状态：

- 一台 Linux 云服务器
- 需要部署网站
- 需要跑 OpenClaw 或网关类服务
- 当前更关心“跑起来没有、占了哪些端口、哪些服务在工作”

### 当前最适合的路线

第一阶段：`Netdata`

第二阶段：`Grafana + OpenTelemetry`

第三阶段：如果全面容器化，再考虑 `Portainer`

## 7. 最推荐的分阶段方案

### 阶段 1：先解决“看机器和端口”

先上 Netdata，主要看：

- CPU
- 内存
- 磁盘
- 网络
- 进程
- socket / 端口活跃度

这是最能立刻带来收益的一步。

### 阶段 2：再解决“看调用链和依赖图”

等你的系统稳定下来后，接入 OpenTelemetry，再用 Grafana 看：

- 服务地图
- Trace
- 请求调用路径
- 下游依赖

### 阶段 3：最后解决“看容器和部署拓扑”

如果后面改成 Docker / Compose 部署，再加 Portainer。

## 8. 你可以把这些工具理解成什么

### Netdata

像是“这台服务器的总控台”。

### Grafana + OpenTelemetry

像是“应用内部请求链路透视图”。

### Portainer

像是“容器化部署的控制面板”。

## 9. 你真正想看的依赖关系，分哪几种

很多人说“依赖关系”，其实混了三种：

### 9.1 端口依赖

例如：

- 网站前端在 3000 端口
- API 在 5000 端口
- 反向代理在 80/443 端口

### 9.2 进程依赖

例如：

- Nginx 转发到 Node 服务
- Node 服务再调用 OpenClaw 网关
- OpenClaw 再访问其他后端能力

### 9.3 请求依赖

例如：

- 浏览器请求首页
- 首页调用 `/api/nav`
- `/api/nav` 又读取某个配置或数据库

真正能画出“服务地图”的，主要是第三类，也就是 OpenTelemetry + Grafana 这种方案。

## 10. 如果你现在想马上开始，最实用的建议

### 最小可行方案

先装：

- Netdata

暂时不急着装：

- Grafana
- OpenTelemetry Collector
- Portainer

### 原因

因为你现在最先需要看清的是：

- 服务是不是起来了
- 哪个端口在监听
- 哪个进程在占资源
- 服务器是不是太吃紧

这些问题，Netdata 最适合马上回答。

## 11. 未来如果要做成“专业一点”的可视化体系

推荐结构会像这样：

- Nginx 或 Caddy：对外入口
- 网站前端服务
- OpenClaw / 网关服务
- OpenTelemetry：采集 traces / metrics / logs
- Grafana：可视化仪表盘和服务地图
- Loki / Tempo / Prometheus：日志、链路、指标后端

这会更像一个完整的可观测性体系。

## 12. 常见误区

### 误区 1：以为一个工具能自动看懂所有依赖

实际上没有哪个工具能在零接入的情况下，完整理解你的业务依赖图。

### 误区 2：只看端口就以为看懂系统

端口只能看到一部分，它看不到完整业务调用链。

### 误区 3：一开始就上太重的方案

如果系统还在快速变化，先上最轻、最直观的方案更划算。

## 13. 针对 Atramenti-Console 的建议结论

### 当前推荐

- 首选：Netdata
- 进阶：Grafana + OpenTelemetry
- 容器化后再补：Portainer

### 不建议现在就做的事

- 为了“看图”立刻把整个系统重构成超重监控体系
- 在还没稳定服务结构前就先堆很多复杂观测组件
